## 1. Введение

Сортировка данных является одной из фундаментальных задач в области информатики. Поразрядная сортировка (Radix Sort)
позволяет достичь линейной сложности относительно количества элементов при фиксированном диапазоне значений. Данная
работа объединяет реализацию и анализ эффективности алгоритма на базе пяти различных технологий: последовательной
(SEQ), OpenMP (OMP), Intel Threading Building Blocks (TBB), механизмов стандартной библиотеки C++ (STL) и гибридной
модели MPI + OpenMP (ALL).

## 2. Постановка задачи

Необходимо реализовать и сравнить алгоритмы поразрядной сортировки (LSD) для вектора 32-битных целых чисел.

- **Входные данные**: Неотсортированный вектор `std::vector<int32_t>`.
- **Выходные данные**: Отсортированный по неубыванию вектор.
- **Ограничения**: Корректная обработка отрицательных чисел, стабильность локальных сортировок, эффективное
  использование вычислительных ресурсов.

## 3. Описание базового алгоритма

Во всех реализациях в качестве основы используется следующий последовательный алгоритм:

1. Определение минимального и максимального элементов.
2. Сдвиг всех значений в положительную область (вычитание минимума, если он меньше нуля) для корректной
   поразрядной обработки.
3. Поразрядный обход (LSD - Least Significant Digit) с использованием стабильной сортировки подсчетом.
4. Обратный сдвиг к исходным значениям.

## 4. Схемы распараллеливания

- **SEQ (Последовательная версия):** Базовый эталон. Все вычисления выполняются последовательно на одном ядре
  процессора без дополнительных накладных расходов на синхронизацию.
- **OMP (OpenMP):** Входной массив делится на $N$ равных частей. Каждый поток независимо выполняет полную локальную
  сортировку своего участка, после чего происходит параллельное древовидное слияние с использованием барьерной
  синхронизации (`#pragma omp barrier`).
- **TBB (Threading Building Blocks):** Используются алгоритмы `tbb::parallel_reduce` (для поиска минимума) и
  `tbb::parallel_for` (для сдвига). Слияние отсортированных блоков организовано в виде дерева задач через
  планировщик TBB, обеспечивающий динамическую балансировку нагрузки.
- **STL (std::async):** Распараллеливание на основе асинхронных задач. Массив делится на блоки, каждый сортируется в
  отдельном `std::async`. Древовидное слияние также выполняется асинхронно парами, синхронизация обеспечивается через
  вызов метода `.get()` объектов `std::future`.
- **ALL (MPI + OpenMP):** Двухуровневая гибридная схема. Корневой процесс (MPI) распределяет блоки данных между узлами
  через `MPI_Scatterv`. Внутри каждого процесса используется OpenMP для параллельной локальной сортировки.
  Отсортированные блоки собираются на нулевом процессе через `MPI_Gatherv`, после чего выполняется финальное слияние.

## 6. Результаты экспериментов

### 6.0 Репродуцируемость

Для запуска используются команды:

1) Конфигурация и билд
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
2) Func тесты
python scripts/run_tests.py --running-type=processes --counts 1 2 3 4
python scripts/run_tests.py --running-type=threads --counts 1 2 3 4
3) Perf тесты
$env:PPC_NUM_THREADS = "4"
$env:PPC_NUM_PROC = "2"
python scripts/run_tests.py --running-type="performance"

### 6.1 Корректность

Для всех технологий проведено функциональное тестирование с использованием **Google Test**. Проверены различные
краевые случаи: инвертированные массивы, массивы с дубликатами, исключительно отрицательные числа, однородные
массивы и граничные значения. Результаты всех пяти реализаций успешно совпали с эталонной сортировкой
`std::ranges::sort`.

## 7. Заключение

В ходе работы были реализованы 5 версий алгоритма поразрядной сортировки. Все параллельные версии (OMP, TBB, STL, ALL)
продемонстрировали существенное ускорение по сравнению с последовательной (SEQ).

- Наилучшую производительность на одном вычислительном узле показала **STL-версия** с максимальным ускорением
  1.89x на 4 потоках.
- Результаты **TBB** и **OMP** оказались крайне близки друг к другу (~1.84x – 1.86x).
- Гибридная версия **ALL (MPI+OpenMP)** продемонстрировала производительность на уровне локальных реализаций. Однако
  ее главное преимущество заключается в возможности горизонтального масштабирования на распределенные системы.
